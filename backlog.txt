[MOV-04] Ordenação de movimentos por conta
  Cada conta pode ter os movimentos ordenados de forma independente.
  O modo de ordenação é guardado no state da conta (campo `ms` em nodeState).

  MODOS:
    date   — Ordenação por data do movimento (comportamento actual, default)
    alpha  — Ordenação alfabética pela descrição
    custom — Ordenação manual por índice (o utilizador arrasta para reordenar)

  IMPLEMENTAÇÃO:
    - Guardar no nodeState do account: `ms=date`, `ms=alpha` ou `ms=idx`
    - Se ms=idx: movements precisam de campo `sort_order` (nova migração)
    - Backend: ordenar no query (date/alpha) ou devolver por sort_order (custom)
    - Frontend:
        - Ícone de ordenação no header de cada AccountTreeNode
        - Dropdown: "By date" / "Alphabetical" / "Custom"
        - Se custom: drag handle nas MovementRow para reordenar
    - Android: deve respeitar o mesmo campo ms do nodeState
  Migração: sort_order em movements (apenas necessário para ms=idx)
  Estimativa: ~3h (date/alpha) + ~1 dia adicional (custom/drag)

[MOV-03] Estado do movimento com histórico de transições
  Cada movimento pode ter um estado/classificador que evolui ao longo do tempo.
  Casos de uso: gestão de pagamentos, despesas pendentes, execução de transferências.

  ESTADOS DEFAULT (todos os planos):
    created   — Criado (estado inicial automático)
    executed  — Executado
    paid      — Pago
    cancelled — Cancelado

  ESTADOS CUSTOM (plano Personal / Pro):
    O utilizador pode criar os seus próprios estados com nome e cor personalizados.
    Exemplo: "Por cobrar", "Em litígio", "Reembolsado", etc.

  HISTÓRICO DE TRANSIÇÕES:
    Cada mudança de estado gera um registo de histórico com timestamp.
    Exemplo para uma despesa:
      2026-04-27 10:00 — Criado
      2026-04-28 09:30 — Executado
      2026-04-30 14:00 — Pago
    O histórico é visível numa secção expandível da MovementRow ou num modal de detalhe.

  IMPLEMENTAÇÃO BACKEND:
    - Nova tabela `movement_states` com: id, name, color, is_default, project_id (null se default), user_id
    - Nova tabela `movement_status_history` com: id, movement_id, state_id, changed_at, changed_by
    - Campo `current_state_id FK` em movements
    - Endpoints:
        GET  /movement-states                    — lista estados disponíveis (default + custom do user)
        POST /movement-states                    — criar estado custom (plano Personal/Pro)
        PUT  /movement-states/{id}               — editar estado custom
        DELETE /movement-states/{id}             — apagar estado custom
        POST /movements/{id}/state               — transição para novo estado (gera histórico)
        GET  /movements/{id}/state-history       — histórico de transições
    - Regra de plano: POST /movement-states retorna 403 com upgradeRequired=true se plano Free

  IMPLEMENTAÇÃO FRONTEND:
    - Badge de estado inline na MovementRow (cor + label), clicável para mudar
    - Dropdown de seleção de estado ao clicar no badge
    - Secção expandível de histórico por baixo da MovementRow (ou modal de detalhe)
    - Página de gestão de estados custom (Settings / Project Settings)
    - Plano Free: mostra os estados default + CTA de upgrade para criar custom

  MIGRAÇÃO:
    V7__Add_Movement_States.sql

  Dependência: [UX-01] (toast para feedback de transições) deve vir antes.
  Estimativa: ~1 semana (backend + frontend + Android).

[TD-04] Header do utilizador desaparece dentro de um projeto
  Ao entrar na vista de projeto (árvore de contas), o header com a informação
  do utilizador (nome, account settings, logout, etc.) deixa de estar visível.
  Deve permanecer sempre presente em todas as vistas da aplicação.
  Causa provável: o layout da página de projeto não inclui o componente de header
  global, ou sobrepõe-no.
  Correção: garantir que o header global é renderizado acima do ProjectView e não
  é removido nem ocultado pelo router ou pelo layout da árvore.

[TD-05] Falta fluxo de registo email/password
  O frontend suporta login por email/password, mas não tem sign-up.
  Resultado: utilizadores novos com email/password não conseguem entrar sem
  pré-criação manual no Firebase Auth (console/admin script/import).
  Correção: adicionar fluxo de criação de conta com
  createUserWithEmailAndPassword(...) e, no sucesso, manter bootstrap para
  sincronizar utilizador na BD da app.


--- FUNCIONALIDADES IDENTIFICADAS NA CONCORRÊNCIA ---
  (baseado em análise de YNAB, Firefly III, Actual Budget, Copilot Money, Toshl)

[COMP-01] Movimentos recorrentes (recurring transactions)
  Agendar um movimento para se repetir automaticamente com uma periodicidade.
  Presente em: YNAB, Firefly III, Actual Budget, Toshl.
  CASOS DE USO: renda mensal, salário, subscrições, seguros, prestações de crédito.

  IMPLEMENTAÇÃO:
    - Campo `recurrenceRule` em movements: null ou string RRULE simplificada
      (ex: "FREQ=MONTHLY;DAY=1", "FREQ=WEEKLY;BYDAY=MO")
    - Periodicidades suportadas: diária, semanal, quinzenal, mensal, anual
    - Cron job backend que gera os movimentos concretos automaticamente
    - UI: toggle "Repetir" no form de criação; badge "↻" na MovementRow
    - Ecrã de gestão de recorrências por projeto (listar, pausar, eliminar)
    - Notificação quando um movimento recorrente é gerado (ver [COMP-06])
  Migração: campo recurrence_rule + recurrence_parent_id em movements
  Dependência: [UX-01] (toast para confirmar que o movimento foi agendado)
  Estimativa: ~1 semana (backend + frontend + Android)

[COMP-02] Metas de poupança (savings goals / piggy banks)
  Definir um objetivo de poupança com valor alvo e prazo, e acompanhar o progresso.
  Presente em: YNAB (Targets), Firefly III (Piggy Banks), Copilot Money.
  CASOS DE USO: fundo de emergência, férias, compra de carro, entrada de casa.

  IMPLEMENTAÇÃO:
    - Nova entidade `SavingsGoal`: id, name, targetAmount, deadline, currentAmount,
      linkedAccountId (opcional), user_id, project_id
    - Pode ser associado a um account da árvore (acumula automaticamente)
      ou gerido manualmente com contribuições
    - UI: barra de progresso visual (ex: €1.200 / €5.000 = 24%)
    - Widget no dashboard com as metas mais próximas do prazo
    - Notificação quando uma meta é atingida
  Migração: nova tabela savings_goals
  Estimativa: ~3 dias (backend + frontend)

[COMP-03] Relatórios e gráficos (reports & charts)
  Visualizações analíticas dos dados financeiros ao longo do tempo.
  Presente em: todos os concorrentes (funcionalidade core).
  TIPOS DE RELATÓRIO:

    a) Evolução de saldo — gráfico de linha por conta ou projeto ao longo do tempo
    b) Despesas por período — barras por semana/mês/trimestre/ano
    c) Comparação de contas — quanto foi para cada ramo da árvore
    d) Net worth — soma de todos os projetos (deve - haver) ao longo do tempo
    e) Relatório de orçamento — referenceValue vs gasto real por conta (%)

  IMPLEMENTAÇÃO:
    - Endpoints de agregação na API por intervalo de datas
    - Biblioteca de gráficos no frontend: Recharts ou Chart.js
    - Filtros: período, projeto, profundidade da árvore, sinal (deve/haver)
    - Export para PNG/PDF (partilha)
  Estimativa: ~1-2 semanas (mais longo pelo design de cada tipo de relatório)

[COMP-04] Partilha de projeto (shared projects / collaborative budgets)
  Partilhar um projeto com outro utilizador para gestão conjunta.
  Presente em: YNAB Together (até 6 pessoas), Splitwise, Toshl (shared wallets).
  CASOS DE USO: casal, família, sócios de negócio, projeto partilhado.

  FUNCIONALIDADES:
    - Convidar utilizador por e-mail para um projeto
    - Permissões: Viewer (leitura) / Editor (leitura+escrita) / Admin (tudo)
    - Feed de atividade partilhado: ver quem criou/editou cada movimento
    - Notificações de atividade de outros membros
    - Plano: apenas planos Personal/Pro (não Free)

  IMPLEMENTAÇÃO:
    - Nova tabela `project_members`: project_id, user_id, role, invited_at, accepted_at
    - Todos os queries de accounts/movements passam a verificar project membership em vez de só user_id
    - Endpoint de convite por e-mail + fluxo de aceitação
  Migração: V8__Add_Project_Members.sql (impacto significativo na segurança — requer auditoria)
  Estimativa: ~2-3 semanas (backend + frontend + Android + segurança)

[COMP-05] Reconciliação de conta (account reconciliation)
  Confirmar que os movimentos registados correspondem ao extrato bancário.
  Presente em: Firefly III, Actual Budget, GnuCash, MoneyMoney.
  FLUXO:
    1. O utilizador abre "Reconciliar conta" e indica o saldo atual no banco
    2. A app mostra a diferença e pede que o utilizador marque os movimentos confirmados
    3. Quando os movimentos marcados fecham o saldo, a reconciliação está completa
    4. Movimentos reconciliados ficam protegidos contra edição acidental

  IMPLEMENTAÇÃO:
    - Campo `reconciledAt` (timestamp) em movements
    - Badge de cadeado na MovementRow para movimentos reconciliados
    - Fluxo de reconciliação no contexto de um account
    - Proteção: editar um movimento reconciliado pede confirmação explícita
  Migração: campo reconciled_at em movements
  Estimativa: ~2-3 dias

[COMP-12] Budget com rollover (saldo não gasto rola para o período seguinte)
  Saldo não utilizado num período é adicionado ao orçamento do período seguinte.
  Presente em: YNAB (filosofia base), Actual Budget, Firefly III (growing auto-budget).
  CONTEXTO DOUBLLY: o referenceValue é um teto; o rollover tornaria o budget

  MODOS:

  IMPLEMENTAÇÃO:

[ARCH-01] Multi-tenancy — VERIFICADO ✓
  As tabelas estão corretamente isoladas por utilizador:
  - accounts: tem user_id FK direta, todas as queries filtram por ela.
  - movements: não tem user_id direta mas todas as queries joinam através
    do account e verificam account.user_id (ex: findByIdAndAccount_User_Id).
  - Nunca existe um findById sem userId nos repositories.
  Não há ação a tomar — arquitetura está correta.

[ARCH-02] Histórico de operações de transformação (consolidate / expand)
  Atualmente os accounts transformados ficam marcados com bstate=consolidated
  e são filtrados da árvore normal. Pretende-se:
  a) Ecrã de "histórico" / auditoria que mostra estes registos ocultos
     com opção de reverter cada operação individualmente.
  b) Operação "flush": apaga definitivamente todos os registos consolidados
     de um projeto (liberta espaço). Importante para tenants com quota de
     armazenamento limitada.
  c) Gestão de quotas de armazenamento por tenant — limite de registos ou
     de MB por subscription plan.
  Dependência: planos de subscription (Free / Personal / Pro) com quotas
  diferentes. Requer UI na página de subscription e lógica no backend.

[ARCH-03] Modelo de pricing sugerido
  Baseado em análise dos concorrentes (YNAB $109/ano, Toshl $40/ano, Copilot $95/ano,
  Monarch $100/ano, Spendee $36/ano). O Doublly não tem bank sync — isso elimina
  o custo de infraestrutura que obriga os outros a cobrar mais. Podemos ser
  genuinamente mais baratos com mais valor único (árvore hierárquica).

  PLANOS SUGERIDOS:

    Free — €0
      - 1 projeto
      - Árvore limitada (ex: 3 níveis de profundidade, máx 20 contas)
      - Estados de movimento: apenas os 4 default (criado/executado/pago/cancelado)
      - Sem histórico de estados
      - Templates de projeto disponíveis (gancho de onboarding)
      - Export: CSV básico

    Personal — ~€3-4/mês (€29-39/ano)
      - Projetos ilimitados
      - Árvore ilimitada (profundidade e contas)
      - Estados custom de movimento + histórico completo
      - Ordenação custom de movimentos
      - Export: CSV + PDF
      - Acesso aos templates de projeto

    Pro — ~€7-8/mês (€59-79/ano)
      - Tudo do Personal, mais:
      - Partilha de projetos (até N utilizadores)
      - API access
      - Templates custom (criar os seus próprios)
      - Export avançado (Excel, relatórios)
      - Suporte prioritário

  VANTAGEM COMPETITIVA DE PRICING:
    Sem bank sync = sem custos de dados bancários = pricing 30-50% abaixo dos
    concorrentes com uma feature core (hierarquia) que eles não têm.

  TRIAL:
    Trial sem cartão de crédito (modelo YNAB) converte melhor que trial com cartão.
    Sugestão: 30 dias de Personal grátis ao registar.

  NOTA: os valores são sugestão — validar com métricas de custo de infraestrutura
  e conversão antes de publicar. Os limites do Free (3 níveis / 20 contas) são
  também apenas sugestão — ajustar conforme testes de onboarding.
